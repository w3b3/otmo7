DOC / 001 — PRINCIPLE 01 COMMUNICATION / RHYTHM
— 01

We don't run scrum dailies.

+ What we're doing instead

Short, unstructured, transparent updates throughout the day to your colleagues — the remote equivalent of bumping into a colleague in the hallway. Asynchronous when possible, synchronous when it actually helps. Driven by what you need to make progress, not by a calendar invite.

DOC / 001 — PRINCIPLE 02 FEEDBACK / CADENCE
— 02

We don't wait for retros to give feedback.

+ What we're doing instead

Asking peers for feedback every week — small or large, process or behavior, communication or pace.

Asking for feedback isn't weakness. It's a signal that you take the peer relationship seriously — and that you believe your colleagues' observations are worth your time.
DOC / 001 — PRINCIPLE 03 TOOLING / NON-NEGOTIABLE
— 03

We don't treat AI tools as optional.

+ What we're doing instead

Using them daily and intensely. This is non-negotiable.

AI tools don't replace engineering judgment — they raise the price of not having any.
DOC / 001 — PRINCIPLE 03.1 AI / HUMANS IN THE LOOP
— 03.1 / continued from 03

Humans in the loop.

When an agent can produce working code in any language or stack, the differentiator is no longer "can you write it." The differentiator is whether you can:

  • Build skills. Repeatable, codified workflows for the actions you do constantly: ticket creation, commits, PRs, smoke checks, incident reports, RCAs.
  • Apply critical thinking to outputs instead of merging the first thing that compiles.
  • Steer the agent toward an architecture that won't collapse in two months.
  • Recognize when the generated solution is technically correct — but solving the wrong problem.
DOC / 001 — PRINCIPLE 03.2 AI / CODE OUTPUT
— 03.2 / continued from 03

We value curiosity and ownership.

+ Curiosity

You actually use the product every day. You care and say something when it's awkward. You ask why a user would do the thing the ticket describes. You don't wait for QA, PM, or anyone to tell you what's broken.

+ Ownership

A module has someone whose name comes up when it's discussed — someone who knows its history, its sharp edges, and where it's going. Not because they were assigned it. Because they took it.

DOC / 001 — PRINCIPLE 03.3 AI / NO SILOS
— 03.3 / continued from 03

No silos for engineers.

Languages and stacks now have low barriers.

Architectural taste, product instinct, and the discipline to interrogate what an agent gives you — those are the high-value assets. That's where your time should go.
DOC / 001 — PRINCIPLE 04 PRAGMATISM / TOOLING DISCIPLINE
— 04

We refine because we must —
not because we want.

Under-utilizing something is not ideal. We build the roads we need to use, if we can maintain them. We buy or pay someone to do it if it makes more sense.

The business will drive most decisions. Your voice will steer a few. Be vocal.

DOC / 001 — PRINCIPLE 05 OWNERSHIP / DIRECTION-SETTING
— 05

We don't have PMs.

— What we're not doing

Outsourcing direction-setting to a product manager who tells engineers what to build.

+ What we're doing instead

We have engineers who own the product.

DOC / 001 — PRINCIPLE 05.1 OWNERSHIP / ARCHITECTURE
— 05.1 / continued from 05

Engineering architecture review is a must.

Evolution of the system is shared and discussed openly — every week. Not optional.

If a system is changing and the team doesn't know how, that's the failure mode we're avoiding.

DOC / 001 — PRINCIPLE 05.2 OWNERSHIP / WHERE IDEAS COME FROM
— 05.2 / continued from 05

Where ideas come from.

Product teams here are composed of engineers delivering value.

20%
Leadership

Touch points. Strategic hints and direction.

80%
Engineers

Driving value through real ownership of their product or module.

The deciding voice on what to build next is, most of the time, yours.
DOC / 001 — PRINCIPLE 06 OBSERVABILITY / SIGNALS
— 06

We don't ship in the dark. We wire for signals.

— What we're not doing

Shipping features and hoping someone tells us if they're broken or unused. Treating dashboards, logs, traces, and metrics as something to add later when there's time.

+ What we're doing instead

Every feature, every module, every service emits something readable from day one. Logs that say what happened. Metrics that say how often and how fast. Traces that say where time was spent. Errors that surface with enough context to act on.

The cultural rule that follows: the absence of a signal is a signal.
DOC / 001 — PRINCIPLE 06.1 OBSERVABILITY / WIRE BEFORE YOU SHIP
— 06.1 / continued from 06

Wire it before you ship it.

This is what makes engineer-driven product decisions (Principle 5) honest.

  • Ownership without signals is just opinion.
  • Curiosity without telemetry is just guessing.
  • Instrumentation is what gives your judgment something real to work with.
DOC / 001 — PRINCIPLE 06.2 OBSERVABILITY / EXPERIMENTATION
— 06.2 / continued from 06

We don't ship and pray. We flag, measure, and decide.

  • Decouple deploy from release. Code goes to production behind feature flags. Exposure is a separate, controlled decision.
  • Ship dark, then light up gradually. Internal first. Slice of users next. Watch the signals. Roll forward or roll back based on what you see.
  • Run experiments where they matter. When two reasonable approaches exist and the answer isn't obvious, set up the comparison and let real usage decide.
  • Decisions are data-backed by default. In architecture review, the first question on any proposed change is "what did the telemetry show?" — not "what do you think?"
This is how the no-PM model survives contact with reality. You replace roadmap intuition with controlled exposure to actual users.
DOC / 001 — PRINCIPLE 07 REVIEW / EVIDENCE
— 07

We don't review PRs by eye. We review with Claude Code.

  • PR descriptions are full and evidence-backed. What changed, why, what was tested, what was verified, screenshots or logs where they help. The description is the artifact future-you will read when something breaks.
  • Reviews happen with Claude Code. Agents catch what tired human eyes miss, and they do it consistently.
  • All systems are accessible via CLI or MCP. If your environment can only be inspected through a GUI, that's a gap to close. Hard debugging cases will always need a human — but the path to that human should be terminal-first.
  • If a human reviews it, they ran it. No reproduction steps? The PR is not ready.

DOC / 001 — PRINCIPLE 08.1 OUTPUT / RESPONSIBILITY
— 08.1 / continued from 08

Surface your contribution.

It is not by accident or chance that we are all here. It must be intentional.

We all win if we all share the same boat — rowing on the same cadence, towards the same end goal.

  • If a day of effort produced no artifact — something went wrong. Either the work wasn't worth it, or the output wasn't captured.
  • If a PR merged but no signal moved — the work isn't done.
  • Both are problems worth surfacing.
